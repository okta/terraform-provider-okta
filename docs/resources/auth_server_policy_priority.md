# Resource: okta_auth_server_policy_priority

Manages the priority ordering of authorization server policies for a single authorization server. Use this resource to guarantee consistent evaluation order when multiple `okta_auth_server_policy` resources are applied in parallel, avoiding the race condition described in GitHub issues [#1198](https://github.com/okta/terraform-provider-okta/issues/1198) and [#2140](https://github.com/okta/terraform-provider-okta/issues/2140).

~> **NOTE:** This resource manages ordering only. It does not create or delete policies. On `destroy`, the policies themselves are preserved; only the managed ordering state is removed.

~> **NOTE ON PRIORITY CONFLICTS:** The `priority` attribute on `okta_auth_server_policy` is required by that resource, but when using `okta_auth_server_policy_priority` the value you set is a placeholder — this resource will overwrite it on every apply. To prevent a perpetual diff, add `lifecycle { ignore_changes = [priority] }` to each `okta_auth_server_policy` resource and treat `okta_auth_server_policy_priority` as the single source of truth for ordering.

## Example Usage

```terraform
resource "okta_auth_server" "example" {
  name        = "example"
  description = "example"
  audiences   = ["api://example"]
}

resource "okta_auth_server_policy" "first" {
  auth_server_id   = okta_auth_server.example.id
  status           = "ACTIVE"
  name             = "first"
  description      = "first policy"
  priority         = 1 # placeholder — managed by okta_auth_server_policy_priority
  client_whitelist = ["ALL_CLIENTS"]

  lifecycle {
    ignore_changes = [priority]
  }
}

resource "okta_auth_server_policy" "second" {
  auth_server_id   = okta_auth_server.example.id
  status           = "ACTIVE"
  name             = "second"
  description      = "second policy"
  priority         = 1 # placeholder — managed by okta_auth_server_policy_priority
  client_whitelist = ["ALL_CLIENTS"]

  lifecycle {
    ignore_changes = [priority]
  }
}

resource "okta_auth_server_policy" "third" {
  auth_server_id   = okta_auth_server.example.id
  status           = "ACTIVE"
  name             = "third"
  description      = "third policy"
  priority         = 1 # placeholder — managed by okta_auth_server_policy_priority
  client_whitelist = ["ALL_CLIENTS"]

  lifecycle {
    ignore_changes = [priority]
  }
}

resource "okta_auth_server_policy_priority" "example" {
  auth_server_id = okta_auth_server.example.id
  priorities = [
    okta_auth_server_policy.first.id,
    okta_auth_server_policy.second.id,
    okta_auth_server_policy.third.id,
  ]
}
```

## Schema

### Required

- `auth_server_id` (String) The ID of the authorization server. Changing this forces a new resource.
- `priorities` (List of String) Ordered list of authorization server policy IDs. The first entry receives priority 1, the second receives priority 2, etc.

### Read-Only

- `id` (String) The ID of this resource (same value as `auth_server_id`).

## Drift Detection

On each `terraform plan`, this resource reads the current priority order from the Okta API and compares it to the configured `priorities` list. If policies have been reordered outside Terraform, the plan will show a diff and apply will restore the configured order.

If a managed policy has been deleted outside Terraform, it will be silently removed from state on the next refresh. The resulting shortened list will appear as a diff on the next `terraform plan`, prompting you to reconcile the configuration.
